Configuration

Hexactyl is configured primarily through the .env file located at the root of the project directory. This page documents every important configuration variable and how to set up integrations.

The .env file controls all core panel behavior. Below is a reference of the most important variables:

Mail Configuration

Hexactyl sends email notifications for account registration, password resets, and server events. Configure your mail provider in the .env file.

Using SMTP (Recommended)

MAIL_MAILER=smtp
MAIL_HOST=smtp.yourdomain.com
MAIL_PORT=587
MAIL_USERNAME=panel@yourdomain.com
MAIL_PASSWORD=your_smtp_password
MAIL_ENCRYPTION=tls
MAIL_FROM="panel@yourdomain.com"
MAIL_FROM_NAME=Hexactyl

Using Mailgun

MAIL_MAILER=mailgun
MAILgun_KEY=your-mailgun-api-key
MAILgun_DOMAIN=mg.yourdomain.com

Test your mail configuration with:

php artisan mail:test youremail@example.com

S3 Backup Storage

Hexactyl supports S3-compatible object storage for automatic server backups. Configure the following in your .env:

AWS_ACCESS_KEY_ID=your-access-key
AWS_SECRET_ACCESS_KEY=your-secret-key
AWS_DEFAULT_REGION=us-east-1
AWS_BUCKET=your-backup-bucket
AWS_ENDPOINT=https://s3.amazonaws.com
AWS_URL=https://your-backup-bucket.s3.amazonaws.com

Once configured, enable backups in the admin panel under Settings → Backups. Users can then create manual or scheduled backups from their server management page.

Supported S3-compatible providers include AWS S3, DigitalOcean Spaces, MinIO, and Backblaze B2.

Docker Daemon Setup

Each node runs the Wings daemon, which manages Docker containers for game servers. Ensure Docker is installed and the current user is in the docker group:

sudo usermod -aG docker $USER
newgrp docker
docker --version

Wings communicates with the panel over HTTPS. On first run, it downloads the auto-configuration from the panel and writes its config to /etc/wings/config.yml.

Security Settings

Hexactyl includes several security features that should be enabled in production:

  • Two-Factor Authentication (2FA) — Users can enable TOTP-based 2FA in their account settings. Admins can enforce 2FA for all users in Settings → Security.
  • Application API Keys — API keys are scoped to specific user accounts. Only create keys for trusted integrations.
  • Session Encryption — Sessions are encrypted using the APP_KEY. Rotate this key periodically by running php artisan key:generate --force and re-deploying.
  • Content Security Policy — The panel sets CSP headers to prevent XSS. Do not disable these headers.

Rate Limiting

Rate limiting is enabled by default to prevent abuse. The following limits are configured in the panel:

Action Limit
Login attempts5 per minute
Password reset requests3 per 15 minutes
API requests (authenticated)60 per minute
API requests (unauthenticated)30 per minute
Server console connections10 per minute

Rate limits are enforced using Redis. If you experience issues, verify Redis is running and accessible.

Caching Configuration

Hexactyl uses Redis for caching, sessions, and queue processing. A properly configured Redis instance significantly improves panel performance.

CACHE_DRIVER=redis
SESSION_DRIVER=redis
QUEUE_DRIVER=redis
REDIS_HOST=127.0.0.1
REDIS_PORT=6379
REDIS_PASSWORD=

If you need to flush the cache after a configuration change:

php artisan cache:clear
php artisan config:clear
php artisan route:clear
php artisan view:clear

For production deployments, it is recommended to run a dedicated Redis instance on each node or use a managed Redis service.

Queue Worker Configuration

Background tasks such as server creation, file backups, and notifications are processed through Redis queues. The queue worker runs as a systemd service:

ExecStart=/usr/bin/php /var/www/hexactyl/artisan queue:work --queue=high,standard,low --sleep=3 --tries=3

Monitor queue health with:

php artisan queue:status
php artisan queue:restart

If tasks are stuck, restart the worker with sudo systemctl restart hexactyl-queue.
